PHASE 4 — BACKEND ENFORCEMENT (CRITICAL)
────────────────────────────

Apply backend permission checks to ALL sensitive routes.

A) Sales
POST /sales:
- require can_create_sale
- if payment_type = cash → require can_cash_sale
- if payment_type = partial → require can_partial_sale
- if payment_type = credit → require can_credit_sale
- if credit/partial and customer is required by flow, validate properly
- cashier/salesperson must use req.user.warehouse_id and req.user.safe_id only
- admin/manager may use explicit selected warehouse/safe from request body

POST /sales/:id/cancel:
- require can_cancel_sale

B) Sales Returns
POST /sales-returns:
- require can_return_sale
- return price must equal original sale price
- warehouse/safe/company/user must be stored correctly

DELETE /sales-returns/:id:
- require can_cancel_sale

C) Purchases
POST /purchases:
- require can_create_purchase
- admin/manager choose warehouse
- cashier/salesperson should not use purchase flow unless explicitly allowed by permission design
- safe rules remain enforced

POST /purchases/:id/cancel:
- require can_cancel_purchase

D) Products
GET /products:
- require can_view_products
PUT/DELETE:
- require can_manage_products

E) Customers
GET /customers:
- require can_view_customers
POST/PUT/DELETE:
- require can_manage_customers

F) Inventory
GET pages:
- require can_view_inventory
Adjustments:
- require can_adjust_inventory

G) Expenses
GET /expenses:
- require can_view_expenses
POST /expenses:
- require can_add_expense
- cashier must use req.user.safe_id automatically

H) Vouchers / shift closing
Receipt vouchers:
- require can_add_receipt_voucher
Payment vouchers:
- require can_add_payment_voucher
Shift close:
- require can_close_shift

I) Reports
All /reports/*:
- require can_view_reports
