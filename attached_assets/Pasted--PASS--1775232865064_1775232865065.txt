لا تعدل أي كود.
نفّذ اختبارًا يدويًا فقط على التعديلات الأخيرة، واكتب النتيجة لكل خطوة بوضوح: PASS أو FAIL، مع سبب مختصر.

مهم:
- لا تعمل refactor
- لا تعمل patch
- لا تغير database schema
- فقط اختبر السلوك الحالي
- لو احتجت إنشاء بيانات اختبار، أنشئ أقل قدر ممكن
- بعد كل اختبار، اذكر:
  1) ماذا فعلت
  2) ما النتيجة الفعلية
  3) ما النتيجة المتوقعة
  4) PASS أو FAIL

اختبارات مطلوبة:

========================
الاختبار 1 — منع البيع بأكثر من المخزون
========================
الهدف:
التأكد أن النظام يرفض إنشاء sale إذا كانت الكمية المطلوبة أكبر من الكمية المتاحة.

الخطوات:
1. اختر أو أنشئ منتجًا له كمية مخزون صغيرة واضحة، مثل 2 أو 3.
2. حاول إنشاء sale لهذا المنتج بكمية أكبر من المتاح، مثل 10.
3. لا تعدل بيانات المنتج يدويًا إلا لو لزم إعداد بسيط للاختبار.
4. سجّل response status وresponse body.
5. بعد ذلك تحقق من:
   - هل تم إنشاء sale أم لا
   - هل كمية المنتج في المخزون بقيت كما هي أم تغيّرت

المتوقع:
- status = 400
- رسالة خطأ عربية واضحة تذكر اسم المنتج والكمية المتاحة والمطلوبة أو معناها
- لا يتم إنشاء sale
- المخزون لا يتغير إطلاقًا

========================
الاختبار 2 — إلزام safe_id في purchases النقدية أو الجزئية
========================
الهدف:
التأكد أن purchase نقدية أو جزئية لا تُقبل بدون safe_id.

الخطوات:
1. جهّز طلب purchase صالح فيه items صحيحة.
2. اجعل payment_type = "cash" أو "partial"
3. اجعل paid_amount > 0
4. أرسل الطلب بدون safe_id أو بقيمة null
5. سجّل response status وresponse body
6. تأكد هل تم إنشاء purchase أم لا

المتوقع:
- status = 400
- رسالة خطأ عربية مثل: "يجب اختيار الخزينة للمدفوعات النقدية أو الجزئية"
- لا يتم إنشاء purchase

ثم نفّذ حالة ثانية للمقارنة:
1. أرسل purchase آجل بالكامل
2. payment_type = "credit"
3. paid_amount = 0
4. بدون safe_id

المتوقع:
- الطلب يمر بشكل طبيعي
- يتم إنشاء purchase بنجاح

========================
الاختبار 3 — login بعد حذف backup trigger
========================
الهدف:
التأكد أن تسجيل الدخول ما زال يعمل، لكن بدون تشغيل backup تلقائي.

الخطوات:
1. نفّذ login بمستخدم صالح
2. راقب:
   - هل login نجح
   - هل يوجد بطء غير طبيعي
   - هل يوجد log أو أثر يدل على تشغيل backup
   - هل تم إنشاء backup record جديد أو ملف backup جديد بعد login فقط
3. إن كان هناك جدول backups أو folder backups، راقب هل حصلت إضافة جديدة بسبب login

المتوقع:
- login ينجح بشكل طبيعي
- لا يتم تشغيل backup بسبب login
- لا يظهر backup جديد سببه login فقط
- لا يتأثر نظام النسخ اليدوي

========================
الاختبار 4 — التحقق من اشتراك الشركة في كل request محمي
========================
الهدف:
التأكد أن token صالح لا يكفي وحده إذا كانت الشركة موقوفة أو منتهية.

الخطوات:
1. سجّل الدخول بمستخدم مرتبط بـ company_id
2. احتفظ بالtoken
3. نفّذ request محمي للتأكد أن كل شيء يعمل أولاً
4. بعد ذلك عطّل الشركة أو اجعل end_date منتهيًا
5. أعد استخدام نفس الـ token في request محمي جديد
6. سجّل status وbody

المتوقع في حالة الشركة الموقوفة:
- status = 403
- رسالة واضحة أن الاشتراك موقوف

المتوقع في حالة الشركة المنتهية:
- status = 403
- رسالة واضحة أن صلاحية الاشتراك انتهت

المتوقع لمستخدم بدون company_id:
- يمر request المحمي بشكل طبيعي

========================
الاختبار 5 — post لفاتورة sales
========================
الهدف:
التأكد أن post ينجح ويحوّل الحالة إلى posted وينشئ القيد المحاسبي.

الخطوات:
1. أنشئ sale صالحة بحالة draft
2. تأكد من حالتها قبل post
3. نفّذ POST /sales/{id}/post
4. راقب:
   - status
   - posting_status بعد العملية
   - هل تم إنشاء journal entry مرتبط بالفاتورة
5. اذكر النتيجة

المتوقع:
- نجاح العملية
- posting_status تصبح posted
- journal entry يُنشأ بشكل صحيح
- لا توجد حالة نصف مكتملة

========================
الاختبار 6 — cancel لفاتورة sales
========================
الهدف:
التأكد أن cancel يعكس الأثر بطريقة صحيحة.

الخطوات:
1. استخدم sale موجودة ومناسبة للإلغاء
2. لو كانت posted، نفّذ cancel
3. راقب:
   - posting_status
   - هل عاد المخزون
   - هل تم عكس أثر الخزينة إن وجد
   - هل تم عكس أثر العميل إن وجد
   - هل تم إنشاء reversal journal entry إذا كانت posted
4. اذكر النتيجة

المتوقع:
- posting_status تصبح cancelled
- المخزون يعود بشكل صحيح
- الأرصدة تُعكس بشكل صحيح
- لو كانت posted، يوجد reversal journal entry

========================
الاختبار 7 — post لفاتورة purchases
========================
الهدف:
التأكد أن post في purchases يعمل طبيعيًا بعد تعديل transaction logic.

الخطوات:
1. أنشئ purchase صالحة
2. تأكد أنها draft قبل post
3. نفّذ POST /purchases/{id}/post
4. راقب:
   - posting_status
   - وجود journal entry
   - عدم وجود أخطاء أو حالة جزئية

المتوقع:
- posting_status تصبح posted
- journal entry يُنشأ بنجاح
- لا توجد بيانات ناقصة أو نصف مكتملة

========================
الاختبار 8 — cancel لفاتورة purchases

الهدف:

الخطوات:

   - posting_status

   - أثر الخزينة
   - أثر العميل/المورد
   - reversal journal entry إذا كانت posted
4. اذكر النتيجة

المتوقع:
- posting_status تصبح cancelled
- المخزون يُعكس بشكل صحيح
- الأرصدة تُعكس بشكل صحيح
- لو كانت posted، يوجد reversal journal entry
